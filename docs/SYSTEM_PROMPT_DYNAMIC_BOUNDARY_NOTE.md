# __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ 学习笔记

本文档总结了 Claude Code 源码中系统提示词分界标记 `__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__` 的作用和设计。

## 一句话总结

**`__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__` 是 Claude Code 客户端源码中，用于分隔系统提示词中**「可全局缓存的静态内容」和「不可缓存的动态内容」**的内部分界标记。它配合 Anthropic API 的 Prompt Caching 功能，实现节省 Token 成本和降低延迟的优化。

---

## 分层缓存架构

### 为什么需要分界？

系统提示词分为两种性质完全不同的内容：

| 位置 | 内容 | 性质 | 是否可缓存 |
|------|------|------|-----------|
| **边界之前** | `System`（基础框架规则）、`Doing tasks`（行为准则）、`Executing actions`（风险处理原则）、`Using your tools`（工具使用规范）、`Tone and style`（输出风格约束）、`Output efficiency`（输出效率要求） | **静态不变**：所有用户、所有会话、所有项目都一样 | ✅ 可全局缓存 |
| **边界之后** | `auto memory`（记忆系统规则）、`Environment`（当前环境信息）、项目特定上下文 | **动态变化**：因用户/会话/项目而异 | ❌ 不可缓存 |

## 完整处理流程

```
┌─────────────────────────────────────────────────────────────────┐
│  1. src/constants/prompts.ts 生成系统提示词                     │
│     [静态规则...] + [__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__] + [动态内容]│
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  2. src/utils/api.ts → splitSysPromptPrefix()                   │
│     根据边界标记切分：找到 marker 后把数组切成两部分              │
│     ┌──────────────────────┐ ┌──────────────────────┐          │
│     │  静态块 → cacheScope:'global' │ │  动态块 → cacheScope:null   │          │
│     └──────────────────────┘ └──────────────────────┘          │
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  3. src/services/api/claude.ts → buildSystemPromptBlocks()      │
│     给每个块加上对应的 cache_control 元数据：                    │
│     - 静态块：{ type: 'ephemeral', scope: 'global' }            │
│     - 动态块：不添加 cache_control                               │
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  4. 发送请求到 Anthropic API                                     │
│     - 只发送分块后的内容，边界标记本身被过滤掉                    │
│     - API 通过每个块的 cache_control 字段判断是否缓存            │
│     - API 不认识也不需要认识 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ │
└─────────────────────────────────────────────────────────────────┘
```

---

## 两层约定关系

| 约定双方 | 约定方式 | 作用 |
|---------|---------|------|
| **Claude Code 代码内部** | `prompts.ts` ↔ `api.ts` 通过 `__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__` | 告诉切分函数"从这里切开" |
| **Claude Code 客户端 ↔ Anthropic API 服务端** | 通过 JSON 字段 `cache_control` | 告诉 API"这个块要不要缓存" |

> ⚠️ **重点澄清**：
> - `__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__` 是**客户端内部标记**，API 看不到也不解析
> - 客户端和 API 之间的缓存约定是通过 `cache_control` 字段完成的

## Prompt Caching 到底是什么？

### 常见混淆澄清

| 概念 | 位置 | 是否有记忆 | 作用 |
|------|------|-----------|------|
| **模型自身推理** | LLM 模型内部 | ❌ 无状态 | 每次请求都是独立推理，确实没有记忆 |
| **Prompt Caching** | Anthropic API 服务端 | ✅ 缓存提示词文本 | 避免重复传输相同内容，节省成本和延迟 |

### 没有缓存 vs 有缓存对比

**没有缓存：**
```
请求 1: [静态 1500 token] + [动态 200 token] → 计费 1700 token
请求 2: [静态 1500 token] + [动态 300 token] → 计费 1800 token
请求 3: [静态 1500 token] + [动态 100 token] → 计费 1600 token
         ↑ 这部分每次完全一样，重复计费浪费钱
```

**有 Prompt Caching：**
```
第一次请求:
  [静态块 1500 token] ← cache_control: {scope: "global"}
  [动态块 200 token]
  → 计费 1700 token（第一次全价，API缓存静态块）

第二次请求:
  [提示使用缓存的静态块] ← 不重复计费
  [新的动态块 300 token] ← 只计费这部分
  → 计费 300 token（节省了 1500 token）
```

## 最终效果

| 对比项 | 没有缓存 | 有缓存（通过边界标记切分） |
|--------|---------|---------------------------|
| **API 计费 Token 数** | 每次完整 1500+ 动态 | 只计动态部分，节省 ~80% |
| **网络传输量** | 每次发完整提示词 | 只发动态部分，减少传输 |
| **API 处理延迟** | 每次都要重新分词 | 分词好的静态块已缓存，更快 |
| **输入给模型的完整 Token** | 完整 | 还是完整（模型拿到的内容不变） |

> 📌 结论：
> - **节省了**：Token 成本 + 网络带宽 + 分词时间 → 省钱降延迟
> - **不节省**：模型推理算力 → 因为模型最终还是拿到完整所有 Token

## 设计亮点总结

1. **关注点分离**：静态/动态分离，只缓存不变的内容
2. **渐进优化**：不改变最终给模型的输入内容，只优化传输和计费
3. **模块清晰**：通过简单的文本标记实现模块间约定，不复杂
4. **成本敏感**：真正为用户着想，实实在在节省 Token 开销
